Twenty CRM para SPALLA/OPERON

Veredito Executivo

Serve, mas não como produto pronto. O Twenty pode ser um bom chassi de CRM operacional para SPALLA/OPERON se a adaptação for feita como uma camada de domínio de mentoria por cima dele: objetos customizados, workflows, ingestão de WhatsApp/calls, busca semântica, dossiê, tarefas e dashboards. Se a ideia for apenas “trocar o Spalla por Twenty e configurar campos”, é uma má ideia.

Twenty puro
   |
   v
CRM flexível para empresas, pessoas, tarefas e workflow

SPALLA/OPERON precisa
   |
   v
CRM de acompanhamento consultivo com memória viva do mentorado
Decisão curta

Minha recomendação é não forkear o core do Twenty no início. Primeiro, validar um piloto com Twenty self-hosted + app customizado + camada externa de ingestão/contexto. Fork de core só depois de provar que a UX nativa do Twenty limita o caso real.

A Pergunta Certa

A pergunta não é:

“O Twenty cadastra cliente, tarefa e nota?”

Isso ele faz.

A pergunta certa é:

“O Twenty aguenta virar o sistema operacional da relação consultiva da Bu-Case, onde cada mentorado tem contexto, histórico, arquivos, WhatsApp, calls, tarefas, dossiê, resultado e rotina anual?”

Analogia simples: Twenty é um galpão modular com energia, prateleiras, catracas e empilhadeiras. SPALLA/OPERON é a fábrica inteira, com esteiras, controle de qualidade, operadores, ordens de produção, estoque, retrabalho e painel de produtividade.

O Diferencial Técnico do Twenty

O Twenty permite criar objetos de domínio com defineObject, adicionar campos a objetos existentes com defineField, criar relações bidirecionais e publicar apps em um servidor Twenty.

Exemplo conceitual:

import { defineObject, FieldType } from 'twenty-sdk/define';

export default defineObject({
  nameSingular: 'dossier',
  namePlural: 'dossiers',
  labelSingular: 'Dossiê',
  labelPlural: 'Dossiês',
  fields: [
    { name: 'stage', label: 'Estágio', type: FieldType.SELECT },
    { name: 'mentorSummary', label: 'Resumo estratégico', type: FieldType.TEXT },
  ],
});

Isso é importante porque SPALLA/OPERON não cabe bem em “campos extras de contato”. Ele precisa de entidades próprias.

Fit-Gap: Twenty vs Necessidade Bu-Case

  • Cadastro de empresas e pessoas.
  • Tarefas com responsável, data, status, rich text e anexos.
  • Notas ligadas a registros.
  • Anexos em registros padrão e customizados.
  • Timeline de atividades.
  • Objetos customizados.
  • Relações entre objetos.
  • Workflows com triggers, schedules, webhooks, HTTP request, code e criação/atualização de registros.
  • Permissões por objeto e campo.
  • Email e calendário.
  • Apps versionáveis como código.
  • Workspace por mentorado: dá para modelar com Company, Person, objetos customizados e layouts, mas “workspace” no Twenty é tenant/ambiente, não ficha operacional.
  • Contexto do mentorado: dá para usar Note/Timeline/Custom Object, mas precisa taxonomia e ingestão.
  • Dossiê: dá para modelar como custom object, mas pipeline, revisão, geração e QA precisam ser construídos.
  • Resultados/health score: dá para persistir como campos/objetos/dashboards, mas cálculo vem de fora ou de workflows customizados.
  • Rotinas anuais: workflows ajudam, mas cadência consultiva precisa de modelagem própria.
  • WhatsApp de grupos como entidade primária.
  • Resumo diário/semanal de grupo WhatsApp.
  • Ingestão de áudio, imagem, arquivo e transcrição com pipeline semântico.
  • Busca semântica estilo baú de arquivos do mentorado.
  • Descarrego com classificação, HITL e criação automática de ações.
  • Dossiê estratégico com produção/revisão/aprovação como state machine madura.
  • Contexto consultivo de longo prazo pronto para usar.
  • Métrica de saúde específica de mentoria.

A Decisão Sobre “Workspace”

O termo “workspace” é perigoso aqui porque significa duas coisas diferentes.

TermoNo desejo do FelipeNo Twenty
Workspace por empresaUma área operacional da conta/clienteGeralmente deveria ser Company + objetos relacionados
Workspace por mentoradoUma ficha-cockpit com tudo da pessoaGeralmente deveria ser Person + MentorshipEnrollment + relações
Workspace TwentyTenant/ambiente com membros, permissões, apps e dadosPesado demais para criar um por mentorado sem necessidade
Heurística

Criar um workspace Twenty por mentorado parece organizado, mas tende a virar bagunça operacional: permissões duplicadas, apps repetidos, automações espalhadas, busca global difícil e visão de carteira quebrada. Só faz sentido se houver isolamento jurídico/contratual extremo entre clientes.

Recomendação

Começar com:

1 workspace Twenty da Bu-Case
   |
   +-- Companies = empresas/clientes
   +-- People = mentorados e stakeholders
   +-- MentorshipEnrollment = vínculo pessoa-programa-ciclo
   +-- Dossier / ContextEvent / ResultMetric = domínio SPALLA

Se no futuro a Bu-Case vender o sistema para outras empresas como SaaS, aí sim considerar:

Twenty workspace por organização compradora
   |
   +-- dentro de cada workspace: empresas, mentorados, jornadas

Onde Twenty Ajuda Muito

1. Modelagem de CRM sem começar do zero

Spalla hoje tem muito código próprio para entidades que todo CRM precisa: pessoa, empresa, tarefa, nota, arquivo, timeline, permissão, visão. Twenty entrega boa parte disso.

Sem Twenty: construir cadastro + tarefa + permissão + views + API + UI
Com Twenty: focar no domínio de mentoria e nas integrações

2. Objetos customizados como código

Isso casa bem com SPALLA/OPERON porque o domínio ainda está evoluindo. Você pode versionar:

  • Dossier
  • ContextEvent
  • MentorshipCycle
  • HealthSnapshot
  • WhatsAppTopic
  • CallInsight
  • KnowledgeFile

3. Workflows para automações simples

Exemplos que cabem:

Checkpoint mensalScheduleCriar Task para consultor revisar mentorado
Call realizadaWebhook externoCriar ContextEvent e follow-up
Health score caiuRecord updatedCriar tarefa de intervenção
Dossiê aprovadoRecord updatedCriar tarefa de apresentação

4. Permissões e governança

Twenty tem roles, permissões por objeto/campo e SSO em plano de organização. Para Bu-Case, isso importa porque alguns dados são sensíveis:

  • financeiro do mentorado;
  • risco de churn;
  • bastidores estratégicos;
  • comentários internos;
  • dados de WhatsApp;
  • dossiês em revisão.

Onde Twenty Pode Atrapalhar

1. Se virar “CRM de vendas com fantasia de mentoria”

O SPALLA não é pipeline de deal. O mentorado não é um lead que avança até compra. Ele é uma pessoa acompanhada por meses/anos, com idas e voltas.

Pipeline de venda:
lead -> oportunidade -> proposta -> fechamento

Mentoria:
entrada -> diagnóstico -> estratégia -> execução -> bloqueio -> ajuste -> resultado -> renovação

2. Se o time tentar enfiar tudo em Notes

Notas livres são ótimas para humanos. Péssimas como fonte operacional principal.

Se tudo vira nota, o sistema perde:

  • status;
  • responsável;
  • origem;
  • confiança da IA;
  • relação com tarefa;
  • relação com dossiê;
  • métrica;
  • revisão humana;
  • busca estruturada.

3. Se forkear cedo demais

Fork parece liberdade, mas cobra imposto todo mês:

Custo do forkConsequência
Atualizações do TwentyMerge difícil
Bugs upstreamVocê vira responsável
Enterprise/licençaPrecisa revisão jurídica
UI customizadaPode quebrar com updates
Time técnicoMais manutenção que produto

Risco de Licença

O Twenty está majoritariamente sob AGPL-3.0 e o arquivo de licença indica arquivos específicos marcados como Enterprise sob licença comercial.

   |

Antes de vender uma adaptação proprietária baseada em fork/modificação do Twenty, precisa revisão jurídica específica da AGPL e da licença Enterprise do Twenty. Tecnicamente é viável; comercialmente e juridicamente precisa validação.

Estratégias Possíveis

Usar Twenty como fonte principal para empresas, pessoas, tarefas, contexto, dossiês e dashboards. Criar app SPALLA dentro dele e manter ingestão externa.

Prós: menos sistema próprio, UI já pronta, objetos e permissões. Contras: exige modelagem cuidadosa e adaptação pesada.

Twenty vira interface CRM e camada operacional. Spalla mantém pipelines críticos: WhatsApp, dossiê, busca semântica, descarrego e integrações.

Prós: menor risco, preserva inteligência existente, valida rápido. Contras: sincronização entre sistemas precisa ser bem desenhada.

Customizar core do Twenty para virar produto SPALLA.

Prós: máxima liberdade de UX. Contras: manutenção alta, risco de licença, updates difíceis.

Recomendação

Opção 2 primeiro. Twenty como front-office e CRM operacional, com motor Spalla/OPERON preservando os pipelines específicos.

Depois de 4 a 8 semanas de piloto, decidir se:

  1. Continua com app Twenty sem fork.
  2. Cria componentes front customizados dentro do app.
  3. Forka trechos específicos.
  4. Abandona Twenty se a UX/arquitetura travar demais.

Decisão Recomendada

Sim, usar Twenty se:

  • A meta for acelerar CRM operacional e UI de backoffice.
  • A Bu-Case aceitar manter uma camada Spalla para WhatsApp, IA, dossiê e busca semântica.
  • O piloto for feito sem fork pesado.
  • A licença for revisada antes de comercializar.
  • O modelo for “um workspace Bu-Case + objetos por empresa/mentorado” no início.

Não, usar Twenty se:

  • A expectativa for substituir o Spalla pronto.
  • O produto final depender de UX extremamente customizada desde o primeiro dia.
  • A equipe não quiser manter sincronização/adaptadores.
  • A licença AGPL for incompatível com a estratégia comercial.
  • WhatsApp/grupo for a interface primária e o CRM for secundário.

Conclusão

Twenty é promissor como base de CRM adaptável para SPALLA/OPERON, mas não é o produto final. O coração do Spalla é o acompanhamento consultivo: WhatsApp, call, contexto, dossiê, plano de ação, health e resultado. Isso precisa continuar como domínio próprio.

O melhor caminho é tratar o Twenty como chassi operacional e o Spalla como motor de inteligência consultiva.

Twenty = lugar onde a equipe trabalha
Spalla = inteligência que entende o mentorado
OPERON = execução rastreável

Combinação boa:
  Twenty UI/CRM
  + Spalla context engine
  + OPERON execution discipline

Parecer: avançar para piloto controlado. Não fazer migração total. Não forkear no começo. Não criar workspace por mentorado. Construir app SPALLA no Twenty, conectar 3 mentorados reais anonimizados, medir se a ficha fica mais útil que o Spalla atual.
